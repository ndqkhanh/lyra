    v1.7.5 "REPL Convergence" — Wave-C Detailed Plan¶

    Created: 2026-04-24, immediately after Wave-B / v1.7.4 shipped. Status: READY FOR EXECUTION (start when v1.7.4 has been reviewed and the user gives the go-ahead). Predecessor: v1.7.4 "Local-First & Provider Polish" — shipped 2026-04-24 with +111 contract tests, 912 / 912 + 2 sandbox-skipped suite green. Master roadmap: 2026-04-24-full-parity-roadmap.md § Wave C. Successor: Wave D — v1.8 "Agentic Backbone".

    Theme¶

    Finish every slash command + keybind cell in §1.1–§1.3 of the parity matrix. After this wave the user-observable REPL surface is at strict parity with claw-code, opencode, and hermes-agent — every cell that was NOW or v1.5/v1.7 in the matrix becomes ✓ shipped (v1.7.5).

    This wave is mostly UI + persistence + one-file-per-feature. No new providers, no agentic plumbing — that's Wave D. Difficulty per feature is low; the wave is large because the count is high.

    Invariants (from master roadmap)¶

    1. TDD-first — every task writes the contract test before the implementation.
    2. No breaking changes — every new keyword arg is keyword-only with a default; every new slash registers via the existing COMMAND_REGISTRY tuple; every new keybind hooks via the existing KEYBINDING_REGISTRY (or equivalent prompt-toolkit binding map).
    3. Evidence before assertion — every ✓ shipped parity-matrix flip names a code symbol AND a test file path.
    4. Optional deps — none in this wave (the only candidate would be image-paste OCR; deferred to Wave F).
    5. Graceful degradation — every feature degrades to a friendly message when its precondition is missing (no git repo for /blame, no subagents running for Ctrl+F, no theme file for /theme custom).
    6. Local-first stays local-first — no Wave-C feature pings the network unless the user already opted in via /web (Wave D).
    7. Plan-mode-default-on invariant — /mode may dispatch to other modes, but plan remains the startup default.
    8. Mock-LLM parity — none of the new slashes invoke an LLM by default (they're meta-commands); when they do (e.g. /ultrareview), they accept a --llm mock route.

    Cross-wave dependencies¶

    • /red-proof needs the TDD state machine — full integration is Wave F. v1.7.5 ships a minimal "run pytest, assert non-zero exit" path so the slash exists and is testable.
    • Ctrl+F (kill all subagents) needs Wave-D's subagent registry to be load-bearing. v1.7.5 ships the binding hook + a "no subagents running" message; the actual kill path lights up automatically once Wave D's SubagentRegistry is a runtime singleton.
    • /pair needs streaming subagents — Wave-C ships the slash as a "hold this terminal pane open while I run" UX, true pair streaming is Wave D.

    Tasks (TDD; ~15 tasks for ~30 features grouped by file boundary)¶

    Each task follows the same five-step shape:

    Step N.1 Write the failing test. Step N.2 Run it RED. Step N.3 Implement. Step N.4 Run it GREEN; run the lyra-cli regression. Step N.5 (Skipped while no git is allowed.) Commit.

    Below is a high-density task list. Each item lists the slash / keybind, the new symbol(s), and the contract test file. Subagent dispatchers should expand each into the canonical 5-step rhythm (see Wave-B Task 1 for the canonical example).

    Task 1: /rewind persistent + /resume real¶

    • Slashes: /rewind, /resume
    • Symbols: upgrade InteractiveSession.rewind_one to also persist a TurnSnapshot to ~/.lyra/sessions/<id>/turns.jsonl; add InteractiveSession.resume_session(session_id) that reads the JSONL back.
    • Test: lyra-cli/tests/test_slash_rewind_resume.py — 4 tests (rewind in-mem still works, rewind persisted, resume restores full state, resume on missing id returns friendly message).

    Task 2: /fork + /rename + /sessions + /export¶

    • Slashes: /fork [new-name], /rename <new-name>, /sessions (list), /export <md|json|jsonl>.
    • Symbols: SessionsStore.fork, .rename, .list, .export_to.
    • Test: lyra-cli/tests/test_slash_session_management.py — 8 tests.

    Task 3: /map real (dependency map)¶

    • Slashes: /map.
    • Symbols: InteractiveSession._cmd_map_text runs a tree-style find . -name "*.py" -path "*/lyra*" and renders an indented ASCII tree (no graphviz dep — keep it stdlib-only).
    • Test: lyra-cli/tests/test_slash_map.py — 3 tests (renders, outside-repo handles cleanly, depth cap honoured).

    Task 4: /blame + /trace + /self¶

    • Slashes: /blame [path] (git blame for current file), /trace (last N HIR events from lyra_core.hir.events ring buffer), /self (renders the current InteractiveSession state as YAML).
    • Symbols: InteractiveSession._cmd_blame_text, _cmd_trace_text, _cmd_self_text; new lyra_core.hir.events.RingBuffer (1024 entries, drop-oldest).
    • Test: lyra-cli/tests/test_slash_blame_trace_self.py — 6 tests.

    Task 5: /budget set + /badges¶

    • Slashes: /budget set <usd> (hard cap with alert at 80% + hard-stop at 100%), /badges (renders earned achievement chips).
    • Symbols: lyra_cli.interactive.budget.{BudgetCap,enforce}; _cmd_badges_text reads from ~/.lyra/badges.json.
    • Test: lyra-cli/tests/test_slash_budget_badges.py — 5 tests.

    Task 6: Keybinds — Ctrl+T, Ctrl+O, Esc Esc, Tab, Alt+T, Alt+M¶

    • Bindings: Ctrl+T (live task panel toggle real), Ctrl+O (verbose tool-call output toggle), Esc Esc (persistent rewind — uses Task 1's persistence), Tab (cycle modes 5-state: build → plan → run → retro → explore → build), Alt+T (deep-think toggle), Alt+M (permission mode toggle).
    • Symbols: new helpers in lyra_cli.interactive.keybinds; each binding flips a bool field on InteractiveSession and emits a toast.
    • Test: lyra-cli/tests/test_keybinds_session_toggles.py — 6 tests.

    Task 7: /mode full dispatcher¶

    • Slashes: /mode <name> (build / plan / run / retro / explore), /mode list, /mode toggle.
    • Symbols: upgrade existing _cmd_mode to dispatch full taxonomy; tie build mode to permission gate (mode-aware, graceful when permission stack absent).
    • Test: lyra-cli/tests/test_slash_mode_full.py — 5 tests.

    Task 8: /handoff (markdown PR description)¶

    • Slashes: /handoff produces a paste-able PR description from the session: title (from latest user prompt), summary (last assistant turn), test plan (from _turns_log containing test_* lines), changelog bullet (from git diff).
    • Symbols: lyra_cli.interactive.handoff.render_handoff(session, *, git_available).
    • Test: lyra-cli/tests/test_slash_handoff.py — 4 tests (renders all sections, no-git fallback, empty-session warning, truncation cap).

    Task 9: /effort interactive slider¶

    • Slashes: /effort opens an arrow-key UI to set the model's reasoning effort (low / medium / high / max — applies to o3-style models and reasoning presets).
    • Symbols: lyra_cli.interactive.effort.{EffortPicker, effort_to_max_completion_tokens}; integrates with the ProviderRouting from v1.7.4 for OpenRouter sort tweaks.
    • Test: lyra-cli/tests/test_slash_effort.py — 4 tests (renders, arrow keys cycle, Enter confirms, sets env var).

    Task 10: /ultrareview + /review + /tdd-gate¶

    • Slashes: /review runs a single post-turn verifier (TDD gate + safety + evidence check). /ultrareview fans out to N reviewer subagents (Wave-D's subagents — guard with mock fan-out here). /tdd-gate on|off|status toggles the TDD gate state.
    • Symbols: InteractiveSession._cmd_review_text, _cmd_ultrareview_text, _cmd_tdd_gate_text; gate stored in ~/.lyra/config.yaml (Task 11).
    • Test: lyra-cli/tests/test_slash_review_tdd_gate.py — 7 tests.

    Task 11: /config + /theme + /skin (alias) — config foundation¶

    • Slashes: /config get <key>, /config set <key> <value>, /config list, /theme [aurora|midnight|paper] (3 themes real, not just the badge), /skin aliases /theme.
    • Symbols: new module lyra_cli.interactive.config with Config.load() / save() / get / set against ~/.lyra/config.yaml; lyra_cli.interactive.themes exports three palettes; _cmd_theme_text switches and persists.
    • Test: lyra-cli/tests/test_slash_config_theme.py — 8 tests.

    Task 12: /vim real toggle (depends on Task 11's config)¶

    • Slashes: /vim on|off|status. Persists to config; the prompt-toolkit input loop swaps key bindings on next prompt.
    • Symbols: InteractiveSession.vim_mode already exists; _cmd_vim_text writes through Config.set("vim_mode", bool); new lyra_cli.interactive.keybinds.vi_bindings() factory.
    • Test: lyra-cli/tests/test_slash_vim.py — 4 tests.

    Task 13: /red-proof minimal¶

    • Slashes: /red-proof <pytest target> runs pytest against the named target, asserts non-zero exit, returns "RED ✓" or "test unexpectedly passed" message.
    • Symbols: lyra_cli.interactive.red_proof.{RedProofResult, run_red_proof}.
    • Test: lyra-cli/tests/test_slash_red_proof.py — 4 tests (red proof passes, false-pass detected, target-not-found, pytest-not-installed graceful message).

    Task 14: /tools + /btw + /pair¶

    • Slashes: /tools lists every registered tool with its schema; /btw <question> opens a side-question with isolated context (uses Wave-D's subagent path when present, falls back to in-line ask); /pair opens a "pair-programming" mode that holds this terminal pane and prints incoming agent thoughts as they arrive (real streaming is Wave D).
    • Symbols: _cmd_tools_text introspects lyra_core.tools.registry; _cmd_btw_text, _cmd_pair_text enter a sub-state of the session.
    • Test: lyra-cli/tests/test_slash_tools_btw_pair.py — 6 tests.

    Task 15: Paste-as-image placeholder + parity matrix + CHANGELOG + Wave-D plan¶

    • Feature: prompt-toolkit bracketed-paste handler that detects base64 PNG / JPEG, writes to ~/.lyra/sessions/<id>/attachments/<n>.<ext>, and substitutes [Image #N] in the prompt. No OCR (deferred to Wave F).
    • Symbols: lyra_cli.interactive.paste.{detect_image_paste, write_attachment}; wired into the input loop.
    • Test: lyra-cli/tests/test_paste_image.py — 4 tests.

    Execution Handoff¶

    Two execution options (same as Wave B):

    1. Subagent-Driven (recommended for parallelism) — Tasks 1, 2, 3, 4, 5 are independent (different files); dispatch them in parallel. Tasks 7–14 mostly touch session.py and must run sequentially. Task 15 is the docs + plan, runs last.

    2. Inline Execution — Same TDD rhythm as Wave B; faster iteration but no parallelism.

    Which approach? Default to subagent-driven for the parallelizable chunk (Tasks 1–5, ~15 minutes wall-clock with 5 parallel subagents), inline for the sequential session.py chunk (Tasks 6–14), inline for the docs (Task 15). Final code-quality reviewer subagent over the full v1.7.5 surface before tagging.

    Stop after Task 15 — present the shipped v1.7.5 release + the Wave-D plan file for user review before Wave D.
