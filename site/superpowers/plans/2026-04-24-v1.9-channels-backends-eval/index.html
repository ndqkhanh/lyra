    Wave E — v1.9 "Channels, Backends, Eval" — Detailed Plan¶

    Date: 2026-04-24 Owner: Lyra harness team Status: drafted at Wave-D ship; ready for subagent-driven execution. Predecessor: 2026-04-24-v1.8-agentic-backbone.md (✓ shipped) Master roadmap: 2026-04-24-full-parity-roadmap.md

    For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement each task. Each task below ships RED → GREEN → REFACTOR with a contract test in the same commit as the implementation.

    1. Goal¶

    Match hermes-agent on every channel adapter and remote terminal backend; build the eval gates that Lyra's TDD-first identity needs; ship the multi-modal pieces (vision, voice) and the wiki / onboarding surfaces. After this wave, Lyra is a strict superset of every reference repo on the channel, backend, and eval axes.

    2. Architecture¶

    15 tasks landing as semver-tagged minor release v1.9.0 "Channels, Backends, Eval". Each task is a single PR / commit batch with its own RED → GREEN → REFACTOR cycle.

    v1.8.0 (✓ shipped, Wave D) ──┐
                             │
                             ├─► v1.9.0 — Wave E — Channels, Backends, Eval
                             │   (~3 weeks, 15 tasks, ~200 new contract tests)
                             │
                             └─► v2.0   — Wave F — Frontier

    3. Invariants¶

    Inherited from the master roadmap §4:

    1. TDD-first. RED → GREEN → REFACTOR per task; contract test in the same commit as the implementation.
    2. No breaking changes. Channel adapters extend lyra_cli.channels.* additively; backends extend harnesses.terminal_backends.* additively; existing public APIs stay stable.
    3. Optional deps. Every channel + backend lands as lyra[<name>] — pip install lyra (no extras) must still launch the REPL.
    4. Graceful degradation. Missing optional dep → FeatureUnavailable with the exact pip install lyra[<extra>] fix command.
    5. Smoke vs unit. Tests that touch live network credentials are @pytest.mark.smoke and gated on LYRA_RUN_SMOKE=1. Paid backends (Modal, Daytona) gate additionally on LYRA_RUN_PAID_SMOKE=1.
    6. Doc surface. Each task appends to CHANGELOG.md v1.9.0 entry, flips the relevant cell in docs/feature-parity.md from NOW / stub → ✓ shipped (v1.9.0), and bumps the Verification snapshot version.
    7. Local-first stays local-first. The local-model path (Ollama / LM Studio / llama.cpp / vLLM / MLX-LM) introduced in Wave B and the local backends introduced here must remain a viable end-to-end path with zero cloud creds.

    4. Task list (15 tasks)¶

    Task 1 — Channel adapter substrate (lyra_cli/channels/base.py)¶

    Theme: the abstract base every channel adapter implements, plus the gateway daemon that fans messages from all configured channels into a single agent-loop scheduler.

    Contract: - class ChannelAdapter(Protocol) — start(), stop(), send(thread_id, text), iter_inbound() -> AsyncIterator[Inbound]. - class Inbound (dataclass): channel, thread_id, user_id, text, attachments, received_at. - class Gateway — owns a list of adapters, manages a per-thread agent-loop session, dispatches inbound to the right session, pushes outbound replies back.

    RED tests (8): - adapter protocol surface (start/stop/send/iter_inbound) - gateway routes inbound to the right session - gateway tolerates an adapter that raises in iter_inbound - gateway emits channel.inbound HIR event - send returns the message id (string) - multiple adapters multiplex in iter_inbound - stop cleanly drains in-flight messages - Inbound round-trips through JSON

    Task 2 — Slack adapter (lyra[slack])¶

    Files: lyra_cli/channels/slack.py, tests/test_slack_adapter_contract.py.

    Optional dep: slack-sdk.

    Behaviour: subscribes to RTM / Socket Mode events; maps each channel + thread to a Inbound; uses chat.postMessage for outbound; honours Sec-Slack-Thread-Ts for threading.

    RED tests (6): auth dance, inbound formatting, outbound threading, attachment passthrough, graceful 429, missing-token error path.

    Task 3 — Discord adapter (lyra[discord])¶

    Mirrors Task 2 against discord.py. 6 RED tests.

    Task 4 — Matrix adapter (lyra[matrix])¶

    Mirrors Task 2 against matrix-nio. 6 RED tests.

    Task 5 — Email + SMS adapters (lyra[email] / lyra[sms])¶

    Bundled because they share the "no real-time, just poll / send" UX.

    • Email: stdlib IMAP + SMTP (aioimaplib for async); supports Gmail, Outlook, Fastmail.
    • SMS: Twilio + Vonage backends behind a SmsBackend Protocol.

    5 RED tests each (10 total).

    Task 6 — Long-tail channel adapters¶

    Feishu / Lark, WeCom, Mattermost, BlueBubbles, WhatsApp (Twilio), Signal (signal-cli REST), OpenWebUI, HomeAssistant, QQBot, DingTalk, generic webhooks. Each gets a single contract test (test_<name>_adapter_contract.py) that asserts the adapter implements the protocol + a happy-path send. 11 tests.

    Task 7 — Modal backend (lyra[modal])¶

    Files: harnesses/terminal_backends/modal.py, tests/test_modal_backend_contract.py.

    ModalBackend(image="python:3.12", cpu=2, memory_mb=4096) — runs AgentLoop.run_conversation on Modal; streams stdout/stderr back to the parent. 6 RED tests; smoke gated on LYRA_RUN_PAID_SMOKE=1.

    Task 8 — SSH backend (lyra[ssh])¶

    Mirrors Task 7 against paramiko / asyncssh. 6 RED tests.

    Task 9 — Daytona + Singularity backends¶

    Bundled. Each ships a contract test + a happy-path smoke. 8 tests total.

    Task 10 — Vision toolkit (lyra[vision])¶

    Files: lyra_core/tools/image_describe.py, lyra_core/tools/image_ocr.py, tests/test_vision_tools_contract.py.

    • image_describe(path) -> dict — calls the active LLM with the paste-as-image attachment.
    • image_ocr(path) -> dict — pytesseract / EasyOCR fallback.
    • Wires into Wave-C paste-as-image: when the user pastes an image, the placeholder [Image #N] resolves through image_describe before the prompt is sent.

    8 RED tests.

    Task 11 — Voice toolkit (lyra[voice])¶

    Files: lyra_cli/voice/{stt.py, tts.py}, tests/test_voice_tools_contract.py.

    • STT via faster-whisper (CPU) or openai-whisper (GPU).
    • TTS via OpenAI audio.speech or pyttsx3 (offline).
    • /voice on|off slash that pipes mic → STT → REPL input and REPL output → TTS.

    6 RED tests + 2 smoke tests.

    Task 12 — Session replay (/replay)¶

    Files: lyra_cli/interactive/replay.py, tests/test_session_replay_contract.py.

    Step through a persisted session event-by-event with a diff overlay showing what changed at each step. 6 RED tests.

    Task 13 — Red-team corpus + safety monitor¶

    Files: tests/red_team/<category>/case_<id>.yaml, lyra_core/safety/monitor.py.

    A small starter corpus (≥30 cases across prompt-injection, secrets-exfil, destructive-cmd, jailbreak categories) + a monitor that runs the corpus on lyra eval safety and reports regressions.

    8 RED tests (4 corpus loaders, 4 monitor behaviour).

    Task 14 — Golden eval corpus + drift gate¶

    Files: tests/golden/<id>.yaml, lyra_core/eval/drift.py, lyra_cli/cli/eval.py (extension).

    A starter set (~20 cases) + lyra eval drift --baseline v1.8.0 that produces a markdown delta report (regressions, new passes, unchanged). 8 RED tests.

    Task 15 — /wiki + /team-onboarding + parity matrix +¶

       CHANGELOG + Wave-F plan
    • /wiki real (file-system markdown by default; Notion + Confluence via lyra[wiki-notion] / lyra[wiki-confluence]).
    • /team-onboarding generates a teammate-targeted ONBOARDING.md from skills + recent sessions.
    • docs/feature-parity.md §5f delta table for v1.9.0.
    • CHANGELOG.md v1.9.0 entry.
    • 2026-04-24-v2.0-frontier.md detailed plan written.

    5. Test target¶

    • ~200 new RED/GREEN contract tests.
    • Combined pytest run target: 1143 → ~1343 green.
    • Smoke-gated tests: ≥30 (channels), ≥10 (backends), ≥5 (vision / voice). Default suite must run without any LYRA_* env vars.

    6. Success criteria¶

    • lyra gateway start --channel slack,discord,email starts the daemon and three channel listeners with no extra wiring.
    • lyra run --backend modal --image python:3.12 --task "…" runs the agent loop on Modal with results streamed back.
    • lyra eval drift --baseline v1.8.0 produces a markdown drift report.
    • lyra voice on enables mic + speakers; lyra voice off restores text-only.
    • The starter red-team corpus runs in <30s and reports zero regressions on the v1.9.0 baseline.

    7. Risks & mitigations¶

    • Live channel credentials. Many channel APIs require working tokens for end-to-end smoke. Mitigation: ship FakeBackend doubles for the unit tier; smoke tests gate on per-channel env vars (LYRA_SLACK_TOKEN, LYRA_DISCORD_TOKEN, …). Default pytest run never touches the live APIs.
    • Paid backends. Modal + Daytona cost real money in smoke. Mitigation: paid smoke gates on LYRA_RUN_PAID_SMOKE=1 separately from LYRA_RUN_SMOKE.
    • Voice / vision install size. faster-whisper + opencv-python bring large wheels. Mitigation: keep them strictly opt-in via lyra[voice] / lyra[vision]; document install size in the README's "Optional features" table.
    • Channel-thread mapping bugs. Cross-channel session bleed-over is a real risk in Slack threads + Discord forums. Mitigation: the Gateway keys sessions on (channel, thread_id) from day one and the contract tests cover the multi-thread case explicitly.

    8. Definition of done¶

    • All 15 tasks GREEN with no skipped or xfail-with-strict tests (smoke skips remain acceptable when their env var is unset).
    • docs/feature-parity.md §5f and §1.4–§1.6 reflect every v1.9.0 cell flip; the Verification snapshot is bumped to v0.7.
    • CHANGELOG.md v1.9.0 entry lists every new module + every test count + the sandbox-only failure carry-over status.
    • 2026-04-24-v2.0-frontier.md exists and is ready for execution.
    • Combined pytest packages/lyra-core packages/lyra-cli -q run is ~1343 passed, 0 regressions.
