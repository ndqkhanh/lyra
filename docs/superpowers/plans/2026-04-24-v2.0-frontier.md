# Wave F detailed plan — `v2.0 "Frontier"`

> Detailed task-by-task plan for the wave that turns Lyra from a
> parity-baseline harness into a frontier research harness.
> Authored 2026-04-24, immediately after Wave E shipped (1339
> tests passing across the repo, only sandbox-only worktree tests
> failing because git is unavailable in the dev sandbox).

## How to read this document

This plan mirrors the structure that worked for Wave B/C/D/E:

* **One section per task.** Each section opens with the task ID
  used in the master roadmap (`f1`…`fN`) so todo list updates
  stay aligned.
* **Goal — Files — Tests — Notes.** The same four headings
  every time, so subagents can be dispatched per-task without a
  per-task briefing rewrite.
* **No git in this wave either.** Continuing the
  `no git allow, just implement it lol` directive: every task
  ends at "tests green." A separate operator handles
  `git add` / `git commit` / `git push`.

The 17 net-new features (§3 of the parity matrix) and 14 frontier
ideas (§4) collapse into the **15 task buckets** below — some
features are merged when they share infrastructure (the meta-
harness + arena, the skill router rewrite + auto-optimizer +
synthesis, etc.) so each task is a coherent landing.

## Task f1 — TDD phase state machine

* **Goal.** First-class, evidence-driven state machine for the
  agent loop: `IDLE → PLAN → RED → GREEN → REFACTOR → SHIP`.
  Strict mode rejects any transition not backed by an artefact
  (test result, applied diff, commit). Lenient mode warns but
  proceeds.
* **Files.**
  * `packages/lyra-core/src/lyra_core/tdd/__init__.py`
  * `packages/lyra-core/src/lyra_core/tdd/state.py` —
    `TDDPhase` enum + `TDDStateMachine` class + `TransitionError`.
  * `packages/lyra-cli/src/lyra_cli/interactive/session.py` —
    surface `session.tdd_phase`; `/phase` slash command renders +
    sets the state.
* **Tests.**
  `packages/lyra-core/tests/test_tdd_state_machine_contract.py`
  pinning every legal transition, every illegal transition, and
  the evidence shape required for each (`PlanArtifact`,
  `RedFailureArtifact`, `GreenPassArtifact`, `ShipArtifact`).
* **Notes.** No external deps. Composes with the safety monitor
  + verifier — TDD-state events flow through the existing HIR
  emitter so dashboards see them for free.

## Task f2 — Cross-channel verifier

* **Goal.** Before any assistant turn surfaces, compare the model
  trace's claims against ground truth. Three channels are
  cross-checked:
  1. The model's own narration ("I edited `foo.py:34`").
  2. The actual git diff applied this turn.
  3. A live filesystem snapshot of the cited file.

  Mismatches trigger a `verifier.miscite` HIR event and (in
  `strict=True`) abort the turn before the user reads it.
* **Files.**
  * `packages/lyra-core/src/lyra_core/verifier/cross_channel.py`
  * `packages/lyra-core/src/lyra_core/verifier/__init__.py` —
    re-export.
* **Tests.**
  `packages/lyra-core/tests/test_cross_channel_verifier_contract.py`
  with golden cases for: (a) accurate citations pass, (b) wrong
  line numbers fail, (c) a non-existent file fails, (d) diff
  trimmed by whitespace still passes.
* **Notes.** Re-uses the existing `unified_diff` helper from the
  replay engine (Wave-E Task 12). When git is missing we degrade
  gracefully — only the trace-vs-FS check runs.

## Task f3 — Refute-or-Promote adversarial stage

* **Goal.** SWE-TRACE-style adversary: a sub-agent attempts to
  refute the proposed solution. If refute succeeds, control loops
  back to PLAN with the counter-evidence appended. If it fails to
  refute after `max_attempts`, the solution is promoted.
* **Files.**
  * `packages/lyra-core/src/lyra_core/loop/refute_or_promote.py`
  * Wires into the existing review/verification pipeline; no slash
    command yet (lives behind `/review --auto` from Task f9).
* **Tests.**
  `packages/lyra-core/tests/test_refute_or_promote_contract.py`
  with stub model callables that always-refute / never-refute /
  refute-once-then-give-up.
* **Notes.** The adversary is a regular `LLMClient` — same surface
  as the rest of `lyra_core`, easy to mock. Token budget controlled
  by the same `EFFORT` env knob as the main loop.

## Task f4 — Rubric Process Reward Model

* **Goal.** Pluggable judge: rubrics under
  `lyra_core/eval/rubrics/<name>.yaml`, judge model called per
  rubric, weighted score returned. Replaces the binary
  pass/fail-only gate of the Wave-E eval corpus (which stays as
  the deterministic floor).
* **Files.**
  * `packages/lyra-core/src/lyra_core/eval/rubrics/__init__.py`
  * `packages/lyra-core/src/lyra_core/eval/prm.py` —
    `Rubric`, `RubricSet`, `prm_score`.
* **Tests.**
  `packages/lyra-core/tests/test_prm_contract.py` with a stub
  judge that returns canned per-rubric scores; pin weighted-
  average maths and per-rubric drift detection.
* **Notes.** Hooked into `/eval-drift` (Task f9) and the
  drift gate from Wave-E.

## Task f5 — Neural Garbage Collection compactor

* **Goal.** Replace the v1.7.3 LLM-driven `/compact` for opt-in
  users. Grow-then-evict context compactor logs every keep/evict
  decision plus the downstream outcome to
  `compactor-outcomes.jsonl` for offline training of a smaller
  classifier.
* **Files.**
  * `packages/lyra-core/src/lyra_core/context/ngc.py`
  * `packages/lyra-cli/src/lyra_cli/interactive/session.py` —
    `/compact ngc` arg.
* **Tests.**
  `packages/lyra-core/tests/test_ngc_compactor_contract.py`:
  fake corpus → fake usage signal → assert evicted entries +
  outcomes JSONL written.
* **Notes.** Outcome log shape:
  `{"turn": int, "kept": [...], "evicted": [...], "outcome": "pass"|"fail"}`.

## Task f6 — Reuse-first hybrid skill router

* **Goal.** Replace the trigger-matcher with BM25 + dense + skill-
  description match. Explicit verdict tags: `MATCH`, `AMBIGUOUS`,
  `NO_MATCH`. The router never silently chooses between siblings.
* **Files.**
  * `packages/lyra-core/src/lyra_core/skills/router.py`
  * `packages/lyra-core/src/lyra_core/skills/__init__.py` — export.
* **Tests.**
  `packages/lyra-core/tests/test_skill_router_contract.py` with
  three skills + ten queries pinning each verdict.
* **Notes.** BM25 via `rank-bm25` (already in scope), dense via
  the in-process embedder shipped in v1.7.5; vector store stays
  optional (`lyra[vector]`).

## Task f7 — Trigger description auto-optimizer

* **Goal.** `lyra skills tune <name>` runs a 5-iteration bounded
  optimisation loop on a 60/40 eval split, persisting the
  highest-scoring trigger description back into the skill file.
* **Files.**
  * `packages/lyra-core/src/lyra_core/skills/tune.py`
  * `packages/lyra-cli/src/lyra_cli/cli.py` — `skills tune` subcmd.
* **Tests.**
  `packages/lyra-core/tests/test_skill_tuner_contract.py` with a
  stub LLM that always proposes one specific edit; pin
  convergence + persistence.
* **Notes.** Hard cap on iterations; cost reported per run.

## Task f9 — `/review --auto` post-turn verifier

* **Goal.** A single `/review --auto` that fans out to: TDD-state

  (Wave-E), refute-or-promote (Task f3), and rubric PRM (Task f4).
  Renders one consolidated report; `--block` flag halts the turn

* **Files.**

    `_cmd_review` extended.
  * `packages/lyra-core/src/lyra_core/loop/auto_review.py` —
    fan-out + merge.
* **Tests.**
  `packages/lyra-cli/tests/test_review_auto_contract.py` with
  every stage stubbed in turn.

## Task f10 — Harness plugins as first-class

* **Goal.** `--harness=dag-teams|single-agent|three-agent|agentless|codeact|meta`
  command-line arg. Each is a registered `HarnessPlugin` with
  `plan` / `execute` / `verify` hooks; the meta plugin is Lyra
  optimising its own harness.
* **Files.**
  * `packages/lyra-core/src/lyra_core/harness/plugin.py` (new
    `HarnessPlugin` ABC + registry).
  * `packages/lyra-core/src/lyra_core/harness/builtins/` (one file
    per built-in plugin).
  * `packages/lyra-cli/src/lyra_cli/cli.py` — `--harness` flag.
* **Tests.**
  `packages/lyra-core/tests/test_harness_plugin_contract.py` —
  registry round-trip, default registration, contract assertions
  on each builtin.
* **Notes.** The `meta` plugin reads from the parity-harness corpus
  (Task f11) and is itself driven by another plugin during eval.

## Task f14 — KLong checkpoint & resume across model generations

* **Goal.** Snapshot every state at compaction boundaries

* **Tests.**
  `packages/lyra-core/tests/test_checkpoint_resume_contract.py`

## Task f15 — Frontier UX bundle: `/split`+`/vote`, `/observe`, ambient budget meter, IDE bridges, plus catch-up grab bag

* **Goal.** Land the remaining frontier items as a single bundle so
  Wave F closes cleanly:
  1. `/split` + `/vote` — fork into N parallel candidates with
     different models; evaluator picks the winner.
  2. `/observe` live panel — Rich `Live` region streaming tool
     calls, token deltas, hook fires.
  3. Ambient budget meter — status-bar widget with $ burn rate +
     >3σ alerts.
  4. Skill diff — semantic delta between proposed + existing
     skill on conflict.
  5. Outcome-weighted memory retrieval.
  6. Mutation testing pass (`mutmut`) wired into `/review --auto`.
  7. Policy-as-code debugger.
  8. Graph-of-sessions DAG view.
  9. Intent embeddings.
  10. Failure-first learning hook.
  11. `/self` dogfood gate.
  12. VS Code / Zed / JetBrains bridge on the v1.7.2 ACP
      scaffold.
* **Files.** One module per item under
  `packages/lyra-cli/src/lyra_cli/frontier/` with explicit
  registration in the slash registry; IDE bridges live in
  `packages/lyra-ide-bridge/` (new sibling package).
* **Tests.** One contract test per item under
  `packages/lyra-cli/tests/frontier/test_<item>_contract.py`. Use
  the standard stub-pattern from Wave-E (no network, no
  external dependency).
* **Notes.** This is intentionally the "long tail" task. The
  smaller items each fit in <150 LOC; tests stay <60 LOC each.
  The big lift is the IDE bridge — schedule it last in the wave.
