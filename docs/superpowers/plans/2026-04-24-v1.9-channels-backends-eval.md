## 1. Goal

Match hermes-agent on every channel adapter and remote terminal
backend; build the eval gates that Lyra's TDD-first identity needs;
ship the multi-modal pieces (vision, voice) and the wiki / onboarding
surfaces. After this wave, Lyra is a strict superset of every
reference repo on the *channel*, *backend*, and *eval* axes.

## 2. Architecture

15 tasks landing as semver-tagged minor release **v1.9.0
"Channels, Backends, Eval"**. Each task is a single PR / commit
batch with its own RED → GREEN → REFACTOR cycle.

```
v1.8.0 (✓ shipped, Wave D) ──┐
                             │
                             ├─► v1.9.0 — Wave E — Channels, Backends, Eval
                             │   (~3 weeks, 15 tasks, ~200 new contract tests)
                             │
                             └─► v2.0   — Wave F — Frontier
```

## 3. Invariants

Inherited from the master roadmap §4:

1. **TDD-first.** RED → GREEN → REFACTOR per task; contract test in
   the same commit as the implementation.
2. **No breaking changes.** Channel adapters extend
   `lyra_cli.channels.*` additively; backends extend
   `harnesses.terminal_backends.*` additively; existing public APIs
   stay stable.
3. **Optional deps.** Every channel + backend lands as
   `lyra[<name>]` — `pip install lyra` (no extras) must still launch
   the REPL.
4. **Graceful degradation.** Missing optional dep →
   `FeatureUnavailable` with the exact `pip install lyra[<extra>]`
   fix command.
5. **Smoke vs unit.** Tests that touch live network credentials are
   `@pytest.mark.smoke` and gated on `LYRA_RUN_SMOKE=1`. Paid
   backends (Modal, Daytona) gate additionally on
   `LYRA_RUN_PAID_SMOKE=1`.
6. **Doc surface.** Each task appends to `CHANGELOG.md` v1.9.0
   entry, flips the relevant cell in `docs/feature-parity.md` from
   `NOW` / stub → `✓ shipped (v1.9.0)`, and bumps the
   `Verification snapshot` version.
7. **Local-first stays local-first.** The local-model path
   (Ollama / LM Studio / llama.cpp / vLLM / MLX-LM) introduced in
   Wave B and the local backends introduced here must remain a
   viable end-to-end path with zero cloud creds.

## 4. Task list (15 tasks)

### Task 1 — Channel adapter substrate (`lyra_cli/channels/base.py`)

**Theme:** the abstract base every channel adapter implements, plus
the gateway daemon that fans messages from all configured channels
into a single agent-loop scheduler.

**Contract:**
- `class ChannelAdapter(Protocol)` — `start()`, `stop()`,
  `send(thread_id, text)`, `iter_inbound() -> AsyncIterator[Inbound]`.
- `class Inbound` (dataclass): `channel`, `thread_id`, `user_id`,
  `text`, `attachments`, `received_at`.
- `class Gateway` — owns a list of adapters, manages a per-thread
  agent-loop session, dispatches inbound to the right session,
  pushes outbound replies back.

**RED tests (8):**
- adapter protocol surface (start/stop/send/iter_inbound)
- gateway routes inbound to the right session
- gateway tolerates an adapter that raises in `iter_inbound`
- gateway emits `channel.inbound` HIR event
- `send` returns the message id (string)
- multiple adapters multiplex in `iter_inbound`
- `stop` cleanly drains in-flight messages
- `Inbound` round-trips through JSON

### Task 2 — Slack adapter (`lyra[slack]`)

**Files:** `lyra_cli/channels/slack.py`,
`tests/test_slack_adapter_contract.py`.

**Optional dep:** `slack-sdk`.

**Behaviour:** subscribes to RTM / Socket Mode events; maps each
channel + thread to a `Inbound`; uses `chat.postMessage` for
outbound; honours `Sec-Slack-Thread-Ts` for threading.

**RED tests (6):** auth dance, inbound formatting, outbound
threading, attachment passthrough, graceful 429, missing-token
error path.

### Task 3 — Discord adapter (`lyra[discord]`)

Mirrors Task 2 against `discord.py`. 6 RED tests.

### Task 4 — Matrix adapter (`lyra[matrix]`)

### Task 5 — Email + SMS adapters (`lyra[email]` / `lyra[sms]`)

Bundled because they share the "no real-time, just poll / send" UX.

- **Email:** stdlib IMAP + SMTP (`aioimaplib` for async); supports

5 RED tests each (10 total).

## 5. Test target

- **~200 new RED/GREEN contract tests.**

- Smoke-gated tests: ≥30 (channels), ≥10 (backends), ≥5 (vision /
  voice). Default suite must run without any `LYRA_*` env vars.

## 6. Success criteria

- `lyra gateway start --channel slack,discord,email` starts the
  daemon and three channel listeners with no extra wiring.
- `lyra run --backend modal --image python:3.12 --task "…"` runs
  the agent loop on Modal with results streamed back.
- `lyra eval drift --baseline v1.8.0` produces a markdown drift
  report.
- `lyra voice on` enables mic + speakers; `lyra voice off` restores
  text-only.
- The starter red-team corpus runs in <30s and reports zero
  regressions on the v1.9.0 baseline.

## 7. Risks & mitigations

- **Live channel credentials.** Many channel APIs require working
  tokens for end-to-end smoke. Mitigation: ship `FakeBackend`
  doubles for the unit tier; smoke tests gate on per-channel env
  vars (`LYRA_SLACK_TOKEN`, `LYRA_DISCORD_TOKEN`, …). Default
  `pytest` run never touches the live APIs.
- **Paid backends.** Modal + Daytona cost real money in smoke.
  Mitigation: paid smoke gates on `LYRA_RUN_PAID_SMOKE=1`
  separately from `LYRA_RUN_SMOKE`.
- **Voice / vision install size.** `faster-whisper` + `opencv-python`
  bring large wheels. Mitigation: keep them strictly opt-in via
  `lyra[voice]` / `lyra[vision]`; document install size in the
  README's "Optional features" table.
- **Channel-thread mapping bugs.** Cross-channel session bleed-over
  is a real risk in Slack threads + Discord forums. Mitigation:
  the `Gateway` keys sessions on `(channel, thread_id)` from day
  one and the contract tests cover the multi-thread case
  explicitly.
